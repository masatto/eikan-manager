# 栄冠ナイン育成管理アプリ

栄冠ナイン(野球育成シミュレーションゲーム)の選手データを管理し、ポジションごとの戦力分析・最適スタメン考案・コンバート(転向)候補の提案を行う静的Webアプリです。

公開URL: https://masatto.github.io/eikan-manager/

## ファイル構成

- `index.html` — 画面構造(分析・選手一覧・選手登録・スタメン・入出力・設定の6タブ)
- `app.js` — 評価計算・画面描画・スタメン最適化ロジック(約1500行)
- `style.css` — ダークモードテーマ(Oswald フォント使用)
- `config.json` — 評価ロジックの重み付け・基準値・特殊能力定義(コード側を変更せずに調整できるよう外出し)

データはブラウザの localStorage に保存される。サーバー側にデータは送られない。入出力タブから JSON 形式でバックアップ出力・読込ができる。

## 用語

- **メイン**: そのポジションを主に守る選手。戦力は 100% として計算する。
- **サブ◎**: サブポジションとして「メインと同様にこなせる」レベル。戦力は 100%(メインと同じ)として計算する。
- **サブ○**: サブポジションとして人並みに守れるレベル。戦力は 70% として計算する。
- **未経験**: メイン・サブのどちらにも指定していないポジション。能力値はあるが経験がないため、戦力は 45% として計算する。

選手登録画面でサブポジションのボタンをタップすると、未選択 → ○(サブ70%) → ◎(メインと同等100%) → 未選択 の3段階で切り替わる。

## 評価ロジック

### ポジション適性(posFit)とポジション戦力(posPower)

各選手の「ポジション適性」(`posFit`)は、そのポジションで重要な能力値(`config.json` の `positionWeights` で定義)の加重平均で決まる。能力値そのものの強さを表す値で、メイン・サブ・未経験の補正は入らない。

「ポジション戦力」(`posPower`)は、ポジション適性に補正率(メイン=1.0 / サブ◎=1.0 / サブ○=0.7 / 未経験=0.45)を掛けた値。実際にそのポジションで起用した場合の戦力を表す。

投手のポジション適性は、能力値の加重平均ではなく投手総合スコア(後述)をそのまま使う。

### ポジション状況の判定(◎○△×)

各ポジションには「実際に試合で守備に入る人数」(`startingCounts`)を設定している。捕手〜遊撃は1人、外野は3人。

判定は「実働人数のうち、最も戦力が低い選手」を基準に行う。外野なら上位3人の中で3番目(最弱)の選手の戦力を見る。「2人が強くても1人が弱ければそこが弱点になる」という考え方。

基準値(`positionThresholds`)は「控えも含めた理想人数(`topCounts`)分の合計」として設定されているため、`topCounts` の人数で割って「1人あたりの基準値」に変換してから判定に使う。

| 判定 | 条件 |
|------|------|
| ◎ | 1人あたり基準値の `excellent` 以上 |
| ○ | `good` 以上 |
| △ | `warning` 以上 |
| × | `warning` 未満 |

### 投手の評価

「3年エース・2年エース・1年エース・中継ぎ候補」の4人構成を理想形とする。`pitcherWeights`(ace / relief / overall)で投手総合スコアを算出し、各学年のエースと中継ぎを選出する(重複なし)。

投手全体の判定は、エース・中継ぎが基準を満たす数で ◎○△× を決定する。

### コンバート候補の提案

△または×判定のポジションについて、能力値的に適性が高い選手をコンバート候補として提案する。

以下の選手は候補から除外する。
- すでにそのポジションをメインまたはサブ◎で守れる選手(サブ○は「◎格上げ」候補として含める)
- コンバート元のポジション(野手)に最低1人が残らなくなる選手
- 現在エース・中継ぎを担っている投手、または抜くと投手層の判定が悪化する投手

### 総合力(overall / mainOverall)

`overallWeights` による加重平均。メインが投手の場合は投手総合スコアを使う。

## スタメン考案

投手を除く野手からハンガリアン法(全体最適の割り当てアルゴリズム)で最適な9人 + DH の10人スタメンを提案する。

### 現実案と理想案

- **現実案**: 現在のメイン・サブ設定の補正込み(`posPower`)で最適化。今すぐ使えるスタメン。
- **理想案**: コンバートを前提に能力値だけ(`posFit`)で最適化。コンバート後の姿。

### 理想案の操作フロー

1. 初期表示: 全員対象で自動最適化した理想案を表示
2. 各ポジションのドロップダウンで別の選手を選ぶ → 仮配置として再最適化に反映
3. 「固定」ボタンを押す → そのポジションが確定、再最適化でも変わらない
4. 「固定中」を再度押す → 固定解除
5. 「固定をリセット」で全ての固定・仮配置を解除

固定は選手IDに紐付いて管理されるため、再最適化で別の外野枠に移動しても固定状態が維持される。

### 理想案の◎○△×

理想案の◎○△×は「この配置が実現した後の評価」を示す。割り当てられた選手をコンバート済み(補正1.0)と仮定して `posFit` で計算するため、現状の◎○△×とは異なる値になる。

### 「このスタメンを実現するために必要な変更」

理想案の下部に、各ポジションに割り当てられた選手のメイン・サブ状況を比較して、コンバートが必要な選手を一覧表示する。

| 表示 | 意味 |
|------|------|
| コンバート必要 | メインでもサブ◎でもサブ○でもないポジションに配置 |
| サブ◎への格上げ | サブ○は持っているがサブ◎に格上げが必要 |
| (表示なし) | メインまたはサブ◎なので変更不要 |

### DH

DH はドロップダウンで選手を選択・固定できる。未固定の場合、守備スタメン外の選手の中からミート×0.4 + パワー×0.4 + チャンス×0.2 が最大の選手を自動選出する。

## 設定タブ

各ポジションの能力値の重み(%)を UI から変更できる。`FIELD_ABIL`(ミート・パワー・走力・肩力・守備力・捕球)と `SPECIAL`(チャンス・対左投手・キャッチャー・ケガしにくさ・盗塁・走塁・送球・回復)の計14項目が全て表示される。

- 0% の項目は評価対象外として扱われる
- 合計が100%でないと保存できない
- 変更した重みは localStorage に保存される
- JSON バックアップ出力時に `weightSettings` として含まれ、インポート時に復元される
- 「リセット」で `config.json` の元の値に戻せる

## 特殊能力

選手登録・編集画面で所持している特殊能力をチェックで登録できる。カテゴリ別(打撃・守備・走塁・勝負強さ・捕手適性・マイナスなど)にグループ化して表示。

| 色 | 種類 |
|----|------|
| 青 | 青特(プラス能力) |
| 黄 | 金特(強力なプラス能力) |
| 赤 | 赤特(マイナス能力) |

現時点ではスコア計算には影響しない(表示のみ)。`config.json` の各能力に `weight` の枠を用意しており、将来の評価反映に対応できる設計になっている。

## config.json の主なキー

| キー | 内容 |
|------|------|
| `rankValues` | G〜S ランクを 1〜8 の数値に変換する対応表 |
| `positionBonus` | メイン/サブ◎/サブ○/未経験の戦力補正率 |
| `topCounts` | ポジションごとの理想人数(控え込み) |
| `startingCounts` | ポジションごとの実働人数 |
| `positionThresholds` | ◎○△× の基準値(`topCounts` 人数分の合計として設定) |
| `positionWeights` | ポジションごとの能力加重平均の重み |
| `overallWeights` | 野手総合力の加重平均の重み |
| `pitcherWeights` | 投手のエース/中継ぎ/総合スコアの重み |
| `pitcherThresholds` | エース/中継ぎとして認める基準スコア |
| `speedScale` / `pitchTypeScale` / `totalBreakScale` | 球速・球種数・総変化量を 1〜8 スコアに変換する際の最小・最大値 |
| `specialAbilities` | 特殊能力の定義一覧(名称・種類・対象・カテゴリ・将来のweight枠) |

## 既知の制約

JSON インポート時は配列であることのみ検証しており、各選手オブジェクトの必須フィールドまでは検証していない。手動で編集した不完全な JSON を読み込むと、表示や計算でエラーが起きる可能性がある。

設定タブで変更した重みと選手データは localStorage に別々に保存されているが、JSON バックアップには両方が含まれるため、バックアップから完全な復元が可能。
